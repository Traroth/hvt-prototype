# Architecture — hvt-prototype

This document records the key design decisions for `hvt-prototype`, along
with the alternatives considered and the reasons they were rejected.

For the full conceptual context, see `docs/hvt-paper-v3.md`.

---

## Purpose

`hvt-prototype` is a proof-of-concept that demonstrates the feasibility of
the HVT stack described in the paper. It is **not** a production
implementation. Its sole objectives are:

1. Compile a Java-defined kernel to SPIR-V via the Beehive SPIR-V Toolkit
2. Dispatch that kernel to a GPU via Vulkan Compute using Project Panama
3. Expose a façade API consistent with the paper's proposed
   `Thread.ofHeterogeneousVirtual()` model
4. Measure and report the speedup over sequential CPU execution

The prototype targets a single GPU (NVIDIA GTX 1080) via Vulkan Compute.

---

## Stack overview

```
User code
    │
    ▼
HVT API façade (Thread.ofHeterogeneousVirtual / HvtMemory / TransferMode)
    │
    ▼
HvtCarrierRegistry — Vulkan device discovery via Panama
    │
    ▼
Kernel dispatcher — submits SPIR-V binary to Vulkan compute queue via Panama
    │
    ▼
SPIR-V binary — generated by KernelCompiler via Beehive SPIR-V Toolkit
    │
    ▼
Vulkan Compute runtime — executes on GPU
```

## Technology choices

### Vulkan Compute over OpenCL for GPU dispatch

**Decision:** GPU dispatch is performed via Vulkan Compute rather than OpenCL,
both accessed through Project Panama (Foreign Function & Memory API, JEP 454,
stable since Java 22).

**Rationale:** NVIDIA's proprietary OpenCL driver does not support
`clCreateProgramWithIL`, the API function required to submit SPIR-V binaries
directly to an OpenCL runtime. Vulkan Compute natively accepts SPIR-V via
`vkCreateShaderModule`, making it the correct and natural path for SPIR-V
dispatch on the GTX 1080. The GTX 1080 supports Vulkan 1.3 with full SPIR-V
support. Vulkan Compute is also architecturally closer to the paper's long-term
vision, as SPIR-V was originally designed as Vulkan's shader intermediate
representation.

**Alternative considered:** OpenCL with SPIR-V via `clCreateProgramWithIL`.

**Reason for rejection:** not supported by NVIDIA proprietary drivers on Pascal
(GTX 1080). Would require falling back to OpenCL C source submission, which
defeats the purpose of using Beehive to generate SPIR-V.

**Alternative considered:** JVOCL or a Vulkan Java wrapper (LWJGL).

**Reason for rejection:** same rationale as for JOCL — Panama is the
direction Java is heading for native interoperability, with no JNI and no C
code required. Pure Java bindings generated by `jextract` are architecturally
closer to what a real HVT implementation would use.

**How:** Vulkan C headers are processed by `jextract` to generate Java
bindings. These generated bindings are committed to the repository under
`src/main/java/fr/dufrenoy/hvt/runtime/vulkan/` and must not be manually
edited.

### Beehive SPIR-V Toolkit for kernel compilation

**Decision:** SPIR-V binaries are generated programmatically from Java
using the Beehive SPIR-V Toolkit (developed at the University of
Manchester, used as the SPIR-V backend for TornadoVM).

**Rationale:** this is the natural Java-native path to SPIR-V — analogous
to ASM for JVM bytecode. It avoids any dependency on an external compiler
toolchain. SPIR-V is the portable intermediate representation for OpenCL
2.1+ and Vulkan compute, ensuring the prototype is not tied to NVIDIA PTX
or any vendor-specific format.

### Java 25 as target version

**Decision:** the prototype targets Java 25.

**Rationale:** Java 25 is the version installed on the development machine
and is the current LTS release. It provides stable Panama (FFM API), the
latest Vector API, and the most current Project Loom APIs. Targeting the
installed version avoids any runtime/compiler mismatch.

### Façade API over real JVM integration

**Decision:** `Thread.ofHeterogeneousVirtual()` is implemented as a façade
that uses a standard `ExecutorService` internally, not a real extension of
the JVM scheduler.

**Rationale:** extending the JVM scheduler to support heterogeneous carriers
requires modifying the JVM itself — outside the scope of a POC. The façade
preserves the proposed public API exactly, demonstrating the programming
model without the JVM integration. This is clearly documented in the paper
and in the prototype README.

### Single AcceleratorType in the prototype

**Decision:** the prototype implements only `AcceleratorType.GPU`. DSP,
FPGA, and NPU support are not implemented.

**Rationale:** demonstrating the stack with one accelerator type is
sufficient to prove feasibility. The API is designed to support multiple
types, but the runtime only routes to GPU in this prototype.

## HVT API façade

### HvtMemory\<T\>

Wraps a data buffer with an explicit `TransferMode`. Backed by an
off-heap `MemorySegment` allocated via Panama.

| TransferMode | Semantics |
|---|---|
| `TO_DEVICE` | Copied from JVM heap to device memory before kernel execution |
| `FROM_DEVICE` | Copied from device memory to JVM heap after kernel execution |
| `DEVICE_ONLY` | Allocated on device, never transferred — for intermediate buffers in kernel chains |

`BIDIRECTIONAL` is intentionally excluded from the prototype (and noted as
a non-goal in the paper).

### HvtThread (façade for Thread.ofHeterogeneousVirtual())

```java
HvtThread t = HvtThread.builder()
    .preferring(AcceleratorType.GPU)
    .fallbackTo(AcceleratorType.CPU)
    .kernel(Kernels::bilinearZoom, srcMem, dstMem, ...)
    .start();
t.join();
int[] result = dstMem.get();
```

Internally uses a single-thread `ExecutorService`. The `preferring()` /
`fallbackTo()` chain is honoured by `HvtCarrierRegistry` at dispatch time.

### HvtErrorBuffer

A device-resident buffer containing a single error code, initialised to
zero. Passed implicitly to every kernel. After kernel execution, the
runtime reads this buffer before transferring output data. If the error
code is non-zero, `HvtKernelException` is thrown and output data is not
transferred to the JVM heap.

Uses `OpAtomicStore` in SPIR-V to allow concurrent writes from multiple
threads without data races.

Limitations (documented honestly in the paper):
- SIMT threads in the same warp continue execution after one thread signals
  an error — partial output corruption may have already occurred
- Error granularity is coarse: which thread triggered the error is unknown
- Hardware faults (device OOM, driver timeout) are handled separately via
  Vulkan error codes

## Async GPU dispatch: VkFence + LockSupport.park()

### Decision

`KernelDispatcher.submit()` does not block the calling thread while the GPU
executes. Instead, it registers a `VkFence` with `GpuCompletionScheduler`
and parks the calling virtual thread with `LockSupport.park()`. The carrier
thread is released during GPU execution and can run other virtual threads.
`GpuCompletionScheduler` is a daemon platform thread that polls all
in-flight fences using `vkWaitForFences(timeout=0)` and calls
`LockSupport.unpark()` when a fence is signalled.

### Rationale

This is the core architectural claim of the HVT paper: a virtual thread
parks on a hardware accelerator exactly as it parks on an I/O socket.
The JVM carrier thread is freed during GPU execution — not blocked on
`vkQueueWaitIdle`. The prototype demonstrates this claim end-to-end:
two concurrent virtual threads can both park on the GPU simultaneously,
with one carrier platform thread serving both.

Without this, the prototype would show the right performance numbers but
miss the key scheduling claim. With it, `KernelDispatcherIntegrationTest
.two_virtual_threads_dispatch_concurrently` directly validates that both
virtual threads park, both get correct results, and both complete correctly
under concurrent access.

### Implementation notes

- `submit()` splits into two synchronized blocks (`beginDispatch` and
  `finishDispatch`) with an unsynchronized park between them. The
  Vulkan device lock is released before parking, allowing concurrent
  dispatches.
- The fence slot (`pFenceSlot`) is allocated in `Arena.ofShared()` because
  `GpuCompletionScheduler` reads it from a different thread.
- `AtomicBoolean done` protects against spurious `LockSupport.park()` wakeups:
  `done` is set to `true` before `unpark()`, and `submit()` loops on
  `while (!done.get())`.
- `DispatchContext` is a private record in `KernelDispatcher` that carries
  all Vulkan handles across the park boundary. Vulkan handles are opaque
  values (addresses) that remain valid regardless of the Arena state.
- The `cmdBuf` pointer slot is reconstructed in `finishDispatch` from a new
  confined arena, because the original pointer slot in Phase 1's temp Arena
  is closed before parking.

### Alternative considered

Polling (`vkQueueWaitIdle` in a loop with `Thread.yield()`).

### Reason for rejection

Polling burns CPU on the carrier thread and does not free it for other
virtual threads. It would demonstrate concurrency correctness but not the
scheduler claim.

## Benchmark target

### Quick benchmark (BilinearZoomBenchmarkTest)

Runs during `mvn test`. Measures wall-clock time for a single GPU dispatch
and a single CPU execution for `bilinearZoom` on a 1920×1080 → 3840×2160
image. Logs `[HVT Benchmark]` and `[HVT Batch Benchmark]` lines.

Results on NVIDIA GTX 1080:
- Single dispatch (full transfer round-trip): GPU ~185 ms vs CPU ~210 ms → ~2.4× speedup
- Batch ×20 (device buffers persistent, transfer amortised): GPU ~13 ms/iter vs CPU ~208 ms/iter → ~15.5× speedup

The single-dispatch figure is dominated by host↔device transfer time. The
batch figure isolates GPU compute throughput.

### JMH benchmark (HvtBenchmark)

Produces `target/benchmarks.jar` via `mvn package`. Runs three benchmarks:
`cpuSingle`, `gpuSingle`, and `gpuBatch` (×20, persistent device buffers).
`@Setup(Level.Trial)` warms up the Vulkan driver with 3 dispatches, then
calls `KernelDispatcher.submitTimed()` and logs a `[Phase breakdown]` line
decomposing single-dispatch time into upload, compute, and download.

`submitTimed()` uses blocking `vkQueueWaitIdle` (not the async path) because
it is a measurement utility, not a concurrency primitive.

Invocation: `java --enable-native-access=ALL-UNNAMED --enable-preview -jar target/benchmarks.jar`
